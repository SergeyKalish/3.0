# Техническое задание: Отчёт №2 «Сводная статистика игрока»

## 1. Название и назначение
**Отчёт №2 — «Сводная статистика игрока» (Player Season Summary)**

Персональный сезонный отчёт по каждому игроку команды «Созвездие 2014».  
Один игрок = один лист А4 (портретная ориентация, PNG).

---

## 2. Общая архитектура модуля

Модуль работает по следующему пайплайну:

1. **Сканирование папки** с `.hkt` файлами.
2. **Фильтрация** — определение «оконченных» матчей.
3. **Сбор данных** — извлечение составов и статистики из оконченных матчей.
4. **Формирование списка игроков** — уникальный список с указанием количества матчей.
5. **UI-выбор** — диалог выбора игрока / группы игроков / всей команды.
6. **Генерация отчётов** — создание PNG-листов для каждого выбранного игрока.
7. **Сохранение** — запись файлов в выбранную папку.

---

## 3. Признак оконченности матча (.hkt)

Матч считается оконченным, если **одновременно** выполнены два условия:

1. В файле присутствует непустой объект `player_shifts_official_timer`.
2. Хотя бы у одного игрока нашей команды есть смена, попадающая в интервал **официального времени с 57:00 по 60:00** (т.е. `official_start` или `official_end` в диапазоне `[3420, 3600]` секунд).

> Если в матче 3 периода по 20 минут ЧИИ, то 57–60 минута — это концовка игры. Наличие смены в этот интервал означает, что матч доигран до конца.

---

## 4. Входные данные и ресурсы

### 4.1. Исходные файлы матчей
- **Папка:** задаётся пользователем через диалог выбора.
- **Формат:** файлы `.hkt` (JSON).
- **Ключевая команда:** `Созвездие 2014` (константа `OUR_TEAM_NAME`).

### 4.2. Фотографии игроков
- **Папка:** `Data/player_photos/`
- **Формат имени файла:** `{id_fhm}-{Фамилия}-{Имя}-№{number}.jpg`
- **Пример:** `8962-Арнаут-Иван-№77.jpg`
- **Размещение в шапке:** по центру, сверху.
- **Fallback:** если фото не найдено — отрисовывать серый прямоугольник-заглушку с иконкой или просто пропускать фото.

### 4.3. База данных игроков (локальный кэш)
- **Файл:** `Data/players_db.json`
- **Создание / обновление:** скрипт `parse_players_db.py`
  - Парсит страницы игроков с сайта `fhmoscow.com/player/{id_fhm}`.
  - При массовом режиме сканирует `.hkt` файлы и собирает данные **только для игроков нашей команды** (`Созвездие 2014`).
  - Если на сайте отсутствует номер или амплуа, используется fallback из полей `rosters.number` и `rosters.role` соответствующего `.hkt` файла.
- **Структура:**
  ```json
  {
    "8962": {
      "name": "Арнаут Иван Данилович",
      "height": "150",
      "weight": "44",
      "hand": "левый",
      "birthdate": "2014-01-20",
      "number": "77",
      "role": "Нападающий"
    }
  }
  ```
- **Поля в шапке отчёта:** Рост, Вес, Хват, Дата рождения.
- Если данных по игроку нет в кэше — поля оставлять пустыми или писать «—».

### 4.4. Логотипы команд
- Используется существующая папка `Data/team_logos/`.
- Логика поиска логотипа — такая же, как в Отчёте №1 (`get_team_logo_path`).

## 5. Структура листа А4 (портрет)

Лист разделён на три зоны по вертикали:

### 5.1. Шапка (верхняя ~25% листа)

**Расположение элементов:**
- **Фотография игрока** — по центру, в верхней части шапки.
- **ФИО игрока** — по центру, под фото (полное имя из `rosters.name`, например «Арнаут Иван»).
- **Текстовый блок** — под ФИО, в две колонки:
  - **Левая колонка:**
    - Команда: `Созвездие 2014`
    - Сезон: `tournament_name` целиком (например, «Первенство г. Москвы по хоккею 2014 г.р. 25/26»)
  - **Правая колонка:**
    - Рост: `... см`
    - Вес: `... кг`
    - Хват: `...`
    - Дата рождения: `DD.MM.YYYY`

> **Примечание:** возраст не выводится, только дата рождения.

### 5.2. Таблица матчей (средняя зона, ~55% листа)

Каждая строка = один матч (один .hkt файл).  
Строки сортируются **по номеру тура по убыванию** (последний тур сверху).

#### Колонки таблицы (17 колонок):

| № | Ключ | Название | Описание |
|---|------|----------|----------|
| 1 | `tour_number` | № тура | Номер тура из матча |
| 2 | `opp_logo` | Лого | Логотип команды-соперника (картинка) |
| 3 | `opp_name` | Соперник | Название команды-соперника (без постфикса « 2014») |
| 4 | `home_away` | Д/Г | «Д» если наш ключ `f-team`, «Г» если `s-team` |
| 5 | `result` | В/П | «В» если наши голы > соперника, «П» иначе |
| 6 | `shifts_count` | СМ | Количество смен игрока в матче |
| 7 | `avg_shift` | СрСм | Среднее время смены в матче (секунды, с `"`) |
| 8 | `total_time` | ВрМ | Общее время на льду в матче (MM:SS) |
| 9 | `powerplay` | Б | Время в большинстве (MM:SS) |
| 10 | `penalty_kill` | М | Время в меньшинстве (MM:SS) |
| 11 | `goals` | Г | Забито голов игроком |
| 12 | `assists` | П | Передачи игрока (1-я и 2-я) |
| 13 | `points` | О | Очки = Г + П |
| 14 | `plus_minus` | +/- | Плюс/минус игрока (только в равных составах) |
| 15 | `penalties` | Ш | Штрафные минуты |
| 16 | `on_ice_gf` | On-Ice GF | Голы нашей команды, когда игрок был на льду (любые ситуации) |
| 17 | `on_ice_ga` | On-Ice GA | Голы соперника, когда игрок был на льду (любые ситуации) |

> **Правила скрытия нулей:** если значение равно 0 (или "0:00", или "0\""), ячейка остаётся пустой (флаг `TABLE_HIDE_ZERO_VALUES = True`, как в Отчёте №1).

### 5.3. Итоговая строка

После всех строк матчей добавляется строка **«ИТОГО»** с агрегатами:

| Колонка | Агрегация |
|---------|-----------|
| № тура | «ИТОГО» |
| Лого | — |
| Соперник | — |
| Д/Г | — |
| В/П | Количество «В» / Количество «П» (например, «8/2») |
| СМ | Сумма |
| СрСм | Среднее по всем матчам (сек) |
| ВрМ | Сумма (MM:SS) |
| Б | Сумма (MM:SS) |
| М | Сумма (MM:SS) |
| Г | Сумма |
| П | Сумма |
| О | Сумма |
| +/- | Сумма |
| Ш | Сумма |
| On-Ice GF | Сумма |
| On-Ice GA | Сумма |

### 5.4. Легенда (нижняя ~15–20% листа)

Расшифровка сокращений колонок таблицы.  
Формат: аббревиатура + пояснение (двумя колонками, как в Отчёте №1).  
Пример: «СМ — Количество смен», «СрСм — Среднее время смены», «On-Ice GF — Голы команды на льду» и т.д.

## 6. Правила расчёта показателей по матчу

Все расчёты выполняются в рамках **одного матча** (одного `.hkt`).  
Логика расчёта идентична Отчёту №1 (лист «Матч»), за исключением специфических правил ниже.

### 6.1. СМ, СрСм, ВрМ, Б, М, Г, П, +/-, Ш
Рассчитываются по той же логике, что и в `report_generator.py` (`_get_cell_text_v2` для `mode='game_on_sheet'`).

### 6.2. Очки (О)
`О = Г + П`

### 6.3. On-Ice GF / On-Ice GA

**Условие попадания гола:** `shift_start < goal_time ≤ shift_end` (как в `+/-`).

- **GF** — количество голов нашей команды (`team == our_team_key`), попадающих в любую смену игрока по правилу выше. Учитываются **все game_mode**, включая пустые ворота (если будут в данных).
- **GA** — количество голов соперника (`team != our_team_key`), попадающих в любую смену игрока по тому же правилу.

> **Важно:** в отличие от `+/-`, On-Ice GF/GA **не фильтрует** равные составы. Считаются абсолютно все голы, произошедшие во время смен игрока.

### 6.4. В/П (Победа/Поражение)
- Считаем общее количество голов нашей команды и соперника по всем голам матча (как `get_final_score()`).
- Наши > Соперника → «В»
- Наши ≤ Соперника → «П» (ничьи невозможны)

### 6.5. Д/Г (Домашняя/Гостевая)
- `our_team_key == 'f-team'` → «Д»
- `our_team_key == 's-team'` → «Г»

---

## 7. Сортировка матчей

Строки в таблице сортируются **по номеру тура по убыванию** (`tour_number` DESC).  
Последний сыгранный тур — на первой строке, самый ранний — внизу.

Если `tour_number` отсутствует или не число — использовать `match_date` как fallback (по убыванию даты).

---

## 8. Выходные файлы

### 8.1. Формат
- PNG, А4 портрет (`210×297` мм).
- Разрешение: 300 DPI (как в Отчёте №1).

### 8.2. Имя файла
Шаблон: `{number}_{Фамилия}__stats.png`

- `number` — номер игрока из `rosters.number`.
- `Фамилия` — первая часть поля `name` (до пробела).

**Пример:** `77_Арнаут__stats.png`

### 8.3. Множественный выбор
- Если выбрано несколько игроков или «вся команда» — генерируется **отдельный PNG-файл для каждого игрока**.
- Все файлы сохраняются в одну выбранную пользователем папку.

## 9. UI / Пользовательский сценарий

1. Пользователь запускает `run_report_2.py`.
2. Появляется диалог выбора **папки** с `.hkt` файлами.
3. Модуль сканирует папку, отбирает оконченные матчи, собирает список игроков.
4. Появляется диалог со списком игроков (с галочками / множественным выбором):
   - Имя игрока
   - Номер
   - Количество матчей в выборке
   - Кнопки: «Выбрать все», «Снять все», «OK», «Отмена».
5. Пользователь выбирает игроков, нажимает OK.
6. Появляется диалог выбора папки для сохранения отчётов.
7. Модуль генерирует файлы и показывает уведомление об успехе с перечнем сохранённых файлов.

## 10. Технические заметки для разработчика

### 10.1. Переиспользование кода
- Использовать существующий `ReportData` как пример для извлечения данных из одного `.hkt`.
- Создать новый класс `PlayerSeasonReportData` (или аналогичный), который:
  - принимает список оконченных `.hkt`, 
  - агрегирует данные по игрокам и матчам.
- Для отрисовки можно создать новый генератор (`PlayerSeasonReportGenerator`), но с переиспользованием:
  - `_get_font()`, `_measure_text_width()`
  - стилей/констант DPI, цветов, шрифтов
  - логики `_draw_table_v2` (при необходимости адаптации под портрет)

### 10.2. Новые сущности
- `SeasonMatchRow` — dataclass для строки таблицы (один матч одного игрока).
- `PlayerSeasonSummary` — dataclass для хранения всех матчей игрока + агрегатов.
- `HKTFileFilter` — логика определения оконченности матча.

### 10.3. Особенности портретной ориентации
- Таблица из 17 колонок широкая. В портрете ширина меньше, чем в альбоме.
- Необходимо либо уменьшить шрифт, либо разрешить перенос текста в ячейках, либо использовать горизонтальную ориентацию для шапки таблицы (вертикальные заголовки), либо комбинацию.
- **Рекомендация:** использовать вертикальные заголовки для узких колонок (как в Отчёте №1) и минимальный шрифт 6–7 pt.

### 10.4. Зависимости
- PyQt5 (диалоги)
- Pillow (генерация PNG)
- Стандартная библиотека (json, os, pathlib)
